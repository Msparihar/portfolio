# PRD: Portfolio v0.3.1 — Desktop OS Theme Overhaul

**Status:** Ready
**Date:** 2026-04-06

---

## Version Goal

Replace the monotone green-on-black hacker aesthetic with a premium neutral dark theme (inspired by Linear, Raycast, and Warp) and add a 11-theme picker in the taskbar system tray, so the desktop OS looks refined and professional rather than vibe-coded.

---

## Background

v0.3.0 shipped the full Desktop OS simulator. The functional shell is solid — windows drag, resize, minimize, the taskbar works, the boot sequence runs. However the visual treatment is `#22c55e` (green-500) used universally on every border, glow, text accent, shadow, and surface tint. This reads as a 2005 "hacker terminal" theme, not a polished tool. The user comparison: "looks vibe coded, not refined like Apple or PostHog."

The CSS variable system (`--dt-*`) was specifically designed for this upgrade. All 14 component files already reference `var(--dt-*)` exclusively. Changing the default theme is primarily a CSS change in `globals.css`. However a code audit found four files with hardcoded green color values that bypass the variable system — these must be migrated to variables as part of this release.

This version also introduces a theme picker component so users can switch to any of 10 color schemes (including the original Emerald green, preserved as an option).

---

## Design Philosophy

The new default "Neutral Dark" theme follows three real design systems as references:

**Linear** — primary reference for the default.
Darkness is native, not inverted light mode. Surfaces differentiate by subtle elevation (slightly lighter = elevated, not darker). Accent color (`#5e6ad2` indigo-violet) appears only on interactive state, selected items, and CTAs — never on borders or background tints. Borders use semi-transparent white (`rgba(255,255,255,0.05–0.08)`), not accent-colored lines.

**Raycast** — secondary reference for tightness and restraint.
Near-monochromatic with a barely-blue-tinted near-black (`#07080a`). Accent (`#FF6363`) used as punctuation only. Text hierarchy through luminance gradation across 5 stops, not hue variation.

**Warp** — warmth reference.
Parchment-warm white text (`#faf9f6`) instead of cold white. Warm near-black background. Almost no accent presence in chrome elements.

**What these systems share that the current theme lacks:**
- No accent color on borders (current: every border is `rgba(34,197,94,0.x)`)
- No green-tinted surface backgrounds (current: `--dt-bg: #080d0a` has green tint, `--dt-surface` surfaces reference green rgba)
- Text hierarchy through neutral gray gradations (current: muted text is `#6b7a6e` — green-tinted)
- Shadows that are pure black-based (current: shadows include green glow component at all times, not just focused state)
- Titlebar/taskbar backgrounds that are neutral dark (current: hardcoded `rgba(24, 36, 26, 0.98)` — the `36, 26` pair is green-tinted)

### 2. Hardcoded Color Migration

**Description:** Four component files bypass the `--dt-*` variable system with hardcoded green-tinted color values. These must be migrated to use the new CSS variables. This is required for the theme picker to work — if `Window.jsx` hardcodes `rgba(34,197,94,0.35)`, switching themes via CSS variables has no effect on the window shadow.

**Files and specific changes:**

#### `D:/Projects/portfolio/src/components/desktop/Window.jsx`

Line 40-41 — `boxShadow` values are hardcoded with green rgba:
```js
// CURRENT (lines 40-41):
boxShadow: isFocused
  ? '0 2px 4px rgba(0,0,0,0.3), 0 8px 24px rgba(0,0,0,0.5), 0 24px 48px rgba(0,0,0,0.4), 0 0 0 1px rgba(34,197,94,0.35), 0 0 80px rgba(34,197,94,0.08), inset 0 0 0 1px rgba(34,197,94,0.05)'
```


### 3. Theme Configuration File

**Description:** Create a JS module that defines all 11 themes as objects. Each theme object contains the complete set of `--dt-*` CSS variable overrides. This file is the single source of truth for all theme colors — both the default theme and the 9 variants.

**File:** `D:/Projects/portfolio/src/config/themes.js`

**Structure:**
```js
export const THEMES = [
  { id: 'neutral', name: 'Neutral', swatch: '#5e6ad2', vars: { /* full variable set */ } },
  { id: 'emerald', name: 'Emerald', swatch: '#22c55e', vars: { /* delta only */ } },
  // ...
];

export const DEFAULT_THEME_ID = 'neutral';
export const THEME_STORAGE_KEY = 'portfolio_theme';
```

**Design decision — full vars vs delta:** Each theme stores only the variables that differ from the neutral base. The theme application function merges base + delta. This keeps the file maintainable and makes it obvious what each theme actually changes.

**The 10 themes with complete accent and surface tint values:**

All themes share the same neutral backgrounds, text, border, scrollbar, and shadow values from the Neutral Dark base. Only accent-family variables and a few tinted surface/scanline values change per theme.

```js
// src/config/themes.js

export const BASE_THEME_VARS = {
  '--dt-accent':               '#5e6ad2',
  '--dt-accent-hover':         '#7170ff',
  // ... all 32 variables from the Neutral Dark definition
};

export const THEMES = [
  {
    id: 'neutral',
    name: 'Neutral',
    swatch: '#5e6ad2',
    vars: {},  // uses base
  },
  {
    id: 'emerald',
    name: 'Emerald',
    swatch: '#22c55e',
    vars: {
      '--dt-accent': '#22c55e',
      '--dt-accent-hover': '#16a34a',
      // ... all emerald overrides
    },
  },
  // ...remaining 8 themes
];

export const DEFAULT_THEME_ID = 'neutral';
export const THEME_STORAGE_KEY = 'portfolio_theme';

/**
 * Apply a theme to the desktop canvas element.
 * Merges base vars with the theme's delta vars, then sets them
 * as inline CSS custom properties on the .desktop-canvas element.
 */
export function applyTheme(canvasEl, themeId) {
  const theme = THEMES.find((t) => t.id === themeId) ?? THEMES[0];
  const merged = { ...BASE_THEME_VARS, ...theme.vars };
  Object.entries(merged).forEach(([prop, value]) => {
    canvasEl.style.setProperty(prop, value);
  });
}
```

### 4. Theme Picker Component

**Description:** A small theme picker UI in the taskbar system tray. Shows a palette icon button that opens a popover with 10 color swatches. Clicking a swatch immediately applies the theme. Persists selection to localStorage. On mount, reads saved theme and applies it.

**Behavior:**
- Palette icon button (monochrome, 16×16, uses a grid/circle glyph or a simple `◑` character)
- Click toggles a popover panel above the taskbar
- Popover shows 10 circular swatches in a single row (2 rows of 5 if row overflow)
- Hovering a swatch shows its name in a tooltip below the swatch
- Active theme swatch has a white ring (`box-shadow: 0 0 0 2px white`)
- Clicking a swatch: calls `applyTheme(canvasEl, themeId)`, saves to `localStorage.setItem(THEME_STORAGE_KEY, themeId)`, closes popover

### 6. localStorage Persistence and Mount Hydration

**Description:** Theme preference is saved to `localStorage` under key `portfolio_theme`. On desktop mount, before first render, the saved theme is applied to prevent a flash of the default theme.

**Implementation in `Desktop.jsx`:**

```js
// After the .desktop-canvas div ref is established, in useEffect:
useEffect(() => {
  const canvasEl = document.querySelector('.desktop-canvas');
  if (!canvasEl) return;
  const saved = localStorage.getItem(THEME_STORAGE_KEY) ?? DEFAULT_THEME_ID;
  applyTheme(canvasEl, saved);
}, []);
```

This runs after mount, so there may be a one-frame flash on cold load. This is acceptable for v0.3.1 — a full SSR/no-flash solution would require injecting a `<script>` in `_document` and is deferred.

**User-facing impact of missing this:** Without the mount hydration, the page always loads with the CSS-default Neutral theme even if the user had selected Emerald. The localStorage read corrects it after ~1 frame. The flash is imperceptible in practice due to the backdrop-filter rendering cost already present.

## Out of Scope

- **Light mode desktop theme** — the OS canvas is dark-only by design. `next-themes` light/dark toggle remains on SEO subpages only.
- **Animated theme transitions** — swapping themes is an instant variable swap. CSS transition on `color`/`background` could make it smoother, but adds complexity and potential jank on low-end devices. Deferred.
- **Flash-of-default-theme prevention via script injection** — would require modifying `layout.js` to inject a blocking inline script. Deferred to a future version if the one-frame flash proves noticeable.
- **System-tray wallpaper cycling** — the existing "Change wallpaper" context menu item cycles wallpaper patterns. This is separate from accent theming and is not touched.
- **Per-app accent overrides** — all app windows share the desktop theme. No per-window theming.
- **Theme export/import** — no custom theme building UI.
- **SEO pages theming** — `/projects`, `/blog`, `/contact` use the standard Tailwind dark/light toggle, completely separate from `--dt-*` variables.
- **Traffic light button colors** — `#ff5f57`, `#febc2e`, `#28c840` are macOS standard. These are not themeable.

1. Subagent 1: `globals.css` — redefine all `--dt-*` variables for Neutral Dark + add 3 new tokens
2. Subagent 2: `themes.js` — create the config file with all 10 theme definitions

**Block B — depends on Block A being done (both new tokens in CSS and themes.js must exist):**
3. Subagent 3: `Window.jsx` + `WindowTitlebar.jsx` — migrate hardcoded colors to variables
4. Subagent 4: `Taskbar.jsx` + `ContextMenu.jsx` + `Desktop.jsx` — migrate hardcoded colors + add ThemePicker slot

**Block C — depends on Block A and `themes.js`:**
5. Subagent 5: `ThemePicker.jsx` — build the theme picker component
6. After ThemePicker exists: wire it into `Taskbar.jsx` (add import + render)

All subagents should be given the complete contents of `globals.css`, `themes.js` (once created), and the specific component file they are modifying.
