# PRD: Portfolio v0.2.0 — Performance & SEO Improvements

**Status:** Ready
**Date:** 2026-04-05

## Version Goal

Eliminate the critical media weight problems causing slow homepage loads and fix the structural SEO gaps (missing H1s, absent og:image file, no canonical URLs) that prevent the site from ranking and sharing correctly on social platforms.

---

## Background

A live-site audit of manishsingh.tech (April 2026) found two independent problem clusters:

**Performance:** 87% of homepage weight (3 MB) is a single GIF (`talent-match.gif`). A second GIF (`linkflow-hero.gif`, 432 KB) is the hero image for the second featured project. Additional GIFs (`threejs-experiment.gif`, `fire.gif`) appear on the projects page. The `<video>` element with WebM/MP4 sources renders identically to GIF but at 5–20% the file size with hardware-accelerated playback.

**SEO:** Every page uses a `<div>` or `<h2>` for its primary heading — no `<h1>` exists anywhere. The root layout metadata references `/images/og-image.png` for `og:image`, but this file does not exist in `public/`, so all social shares produce blank previews. Individual page metadata objects do not set canonical URLs. Blog posts are pure outbound links to `blog.futuresmart.ai` with no on-domain content, so all blog SEO signals flow away from the portfolio domain.

**What is already working and must not regress:**
- TTFB ~200ms, homepage load ~370ms
- JS bundle is small, SSR is functioning
- 13/16 images on /projects are lazy-loaded
- All images have alt text
- 3 JSON-LD structured data scripts (Person, Website, ProfilePage)
- 7 resource preloads configured
- ISR with 24h revalidation on the homepage

## Features

### 1. Convert Featured Project GIFs to Video

**Problem:** `talent-match.gif` (3 MB) is loaded as the hero image of the first featured project card on the homepage. `linkflow-hero.gif` (432 KB) is the second featured project hero. Together they are ~3.5 MB on first paint.

**Solution:** Convert each GIF to two video formats (WebM for modern browsers, MP4 as fallback) and replace `<Image unoptimized>` with a `<video autoplay muted loop playsinline>` element wherever these GIFs appear.

**Implementation details:**
- Conversion tooling: use `ffmpeg` locally. Example:
  ```
  ffmpeg -i talent-match.gif -c:v libvpx-vp9 -b:v 0 -crf 30 talent-match.webm
  ffmpeg -i talent-match.gif -c:v libx264 -pix_fmt yuv420p talent-match.mp4
  ```
- Extract poster frames (static fallback for `prefers-reduced-motion`):
  ```
  ffmpeg -i talent-match.gif -ss 00:00:00 -vframes 1 talent-match-poster.png
  ```
- Output files go to `public/images/` alongside the originals.
- The `<video>` element must have: `autoplay`, `muted`, `loop`, `playsinline`, `aria-label` matching the former `alt` attribute, and a `prefers-reduced-motion` media query that pauses playback.
- Create a reusable `<ProjectVideo>` component (`src/components/ProjectVideo.jsx`) that handles: autoplay, muted, loop, playsinline, aria-label, poster, and prefers-reduced-motion pause logic. Use this in all three rendering surfaces instead of duplicating video logic.
- The `FeaturedProjectCard` component currently detects GIFs with `imageSrc.endsWith('.gif')` and passes `unoptimized` to `<Image>`. This conditional must be extended to render `<ProjectVideo>` instead when the source is a `.webm` or `.mp4`.
- Update `portfolio.json` entries for LinkFlow and Talent Match to reference the `.webm` source (Next.js `<Image>` is not used for video).
- The `ProjectsInteractive` and `ProjectGrid` components on `/projects` also render project images. GIF references there (including `threejs-experiment.gif` and `fire.gif`) must receive the same treatment.

**Reduced-motion handling:** Users with `prefers-reduced-motion: reduce` should see a static poster frame, not an autoplaying video. Add `poster` attribute pointing to a static screenshot and pause the video via CSS: `@media (prefers-reduced-motion: reduce) { video { animation: none; } }` combined with a `useEffect` that calls `video.pause()`.

**Files affected:**
- `public/images/talent-match.webm` (new)
- `public/images/talent-match.mp4` (new)
- `public/images/linkflow-hero.webm` (new)
- `public/images/linkflow-hero.mp4` (new)
- `public/images/threejs-experiment.webm` (new)
- `public/images/threejs-experiment.mp4` (new)
- `public/images/fire.webm` (new)
- `public/images/fire.mp4` (new)
- `src/config/portfolio.json` — update `image` fields for all four projects from `.gif` to `.webm`
- `src/components/FeaturedProjects.jsx` — replace `<Image unoptimized>` with `<video>` for video sources
- `src/components/projects/ProjectGrid.jsx` — same replacement
- `src/components/projects/ProjectModalContent.jsx` — same replacement (modal also shows project images)

**User impact:** Homepage goes from ~3.5 MB to an estimated 200–400 KB for the featured project media. No visual change — videos loop silently, indistinguishable from GIFs. Accessibility: `aria-label` on `<video>`, poster frame for reduced-motion users.

### 2. Add H1 to Every Page

**Problem:** Every page uses `<div>` or `<h2>` for its primary heading. Google's guidelines and web accessibility standards require exactly one `<h1>` per page. The heading hierarchy is broken on all four routes.

**Current state per page:**
- `/` (home): No heading element at all. The terminal "boot sequence" is all `<div>` and `<span>`.
- `/projects`: Primary heading is `<div className="text-lg sm:text-xl md:text-2xl font-bold">$ ls projects/</div>`.
- `/blog`: Same pattern — `<div>` styled as heading.
- `/contact`: Same pattern — `<div>` styled as heading.

**Solution:** Replace the primary heading `<div>` on each page with an `<h1>` that preserves all existing classes. Add a visually-hidden `<h1>` on the homepage where the terminal UI makes a visible heading awkward.

**Implementation details:**

For `/projects`, `/blog`, `/contact`: the `<div className="text-lg sm:text-xl md:text-2xl font-bold text-foreground">` inside the header block becomes `<h1 className="text-lg sm:text-xl md:text-2xl font-bold text-foreground">`. The inner spans remain unchanged. This is a one-character tag change per page — no visual impact.

For `/` (home): The boot sequence renders inside `TerminalContext.jsx` as `<div className="font-mono text-sm space-y-1">`. Insert a visually-hidden `<h1>` before the terminal component in `page.jsx`:
```jsx
<h1 className="sr-only">Manish Singh Parihar — Full Stack & AI Engineer Portfolio</h1>
```
Use `sr-only` (already in Tailwind globals via `@layer utilities`). This is invisible but present in the DOM for crawlers and screen readers.

**Files affected:**
- `src/app/page.jsx` — add `<h1 className="sr-only">` before `<Terminal />`
- `src/app/projects/page.js` — change heading `<div>` to `<h1>`
- `src/app/blog/page.jsx` — change heading `<div>` to `<h1>`
- `src/app/contact/page.jsx` — change heading `<div>` to `<h1>`

**User impact:** No visual change on any page. Screen readers announce the page title correctly. Search engines can identify the primary topic of each page.

### 3. Create the Missing og:image File

**Problem:** `src/app/layout.jsx` and `src/app/metadata.js` both reference `'/images/og-image.png'` in the `openGraph.images` and `twitter.images` arrays. This file does not exist at `public/images/og-image.png`. Every social share of any page on the site produces a blank preview card.

**Solution:** Create a 1200×630px OG image that matches the terminal aesthetic and place it at `public/images/og-image.png`.

**Content:** Black background, green terminal font, displays:
```
Manish Singh Parihar
Full Stack & AI Engineer
manishsingh.tech
```
With the terminal cursor and `$` prompt styling.

**Creation options (choose one):**
- **Option A (recommended):** Use a Next.js `opengraph-image.jsx` file at `src/app/opengraph-image.jsx` with the `ImageResponse` API. This auto-generates at build time and Next.js handles the route. Remove the static `/images/og-image.png` references and let Next.js manage the URL.
- **Option B (simpler):** Design the image in Figma or any tool and export as PNG to `public/images/og-image.png`. No code change needed, references already exist.

**Recommendation:** Option B is faster for v0.2.0. Option A (dynamic generation with `ImageResponse`) can be a v0.3.0 improvement that auto-updates if name/title changes in `portfolio.json`.

For Option B, page-level og:image overrides are also needed on `/projects`, `/blog`, `/contact` — they currently inherit the root og:image. This is acceptable for v0.2.0 since a single shared og:image is standard for portfolio sites.

**Files affected (Option B):**
- `public/images/og-image.png` (new, 1200×630)

**Files affected (Option A — alternative):**
- `src/app/opengraph-image.jsx` (new)
- `src/app/layout.jsx` — remove static `images` array from `openGraph` and `twitter`
- `src/app/metadata.js` — remove static `images` array from `openGraph` and `twitter`

**User impact:** Every social share (LinkedIn, Twitter/X, Discord, Slack, WhatsApp) shows a branded preview card instead of a blank box. This is the highest-visibility SEO issue on the site.

### 4. Add Canonical URLs to All Pages

**Problem:** No page specifies a canonical URL. Without canonicals, search engines may index both `http://` and `https://`, `www.` and non-`www.`, and trailing-slash variants as separate pages, splitting link equity.

**Solution:** Add `alternates.canonical` to the metadata export of every page. Next.js 15 App Router supports this natively — it emits `<link rel="canonical" href="...">` in the `<head>`.

**Implementation details:**

Root layout (`src/app/layout.jsx`) already has `metadataBase: new URL('https://manishsingh.tech')`. Do NOT add canonical to root layout — it conflicts with page-level canonicals. Only add `alternates.canonical` to individual page metadata exports:

- `src/app/page.jsx`: `alternates: { canonical: '/' }`
- `src/app/projects/page.js`: `alternates: { canonical: '/projects' }`
- `src/app/blog/page.jsx`: `alternates: { canonical: '/blog' }`
- `src/app/contact/page.jsx`: `alternates: { canonical: '/contact' }`

Because `metadataBase` is set, Next.js resolves relative paths to full URLs automatically.

**Files affected:**
- `src/app/page.jsx` — add `alternates.canonical` to page metadata
- `src/app/projects/page.js` — add `alternates.canonical`
- `src/app/blog/page.jsx` — add `alternates.canonical`
- `src/app/contact/page.jsx` — add `alternates.canonical`

**User impact:** Not visible. Search engines consolidate signals to the canonical URL, improving ranking consistency.

### 5. Add Preconnect Hints for GitHub API

**Problem:** GitHub contributions data is fetched server-side via ISR, but any client-side refetch (or the `GithubContributions` dynamic import) connects to `github.com` / `api.github.com` without a preconnect hint. The existing resource hints in `layout.jsx` use only `dns-prefetch`, not `preconnect`, which is weaker.

**Solution:** Upgrade the GitHub domain hints to `preconnect` and add `api.github.com`.

**Implementation details:**

In `src/app/layout.jsx`, replace:
```html
<link rel="dns-prefetch" href="//blog.futuresmart.ai" />
<link rel="dns-prefetch" href="//images.unsplash.com" />
<link rel="dns-prefetch" href="//res.cloudinary.com" />
```
with:
```html
<link rel="preconnect" href="https://api.github.com" />
<link rel="preconnect" href="https://avatars.githubusercontent.com" />
<link rel="dns-prefetch" href="//blog.futuresmart.ai" />
<link rel="dns-prefetch" href="//images.unsplash.com" />
<link rel="dns-prefetch" href="//res.cloudinary.com" />
```

`preconnect` vs `dns-prefetch`: preconnect performs DNS + TCP + TLS upfront, eliminating ~200ms from the first GitHub API call. Only add preconnect for origins used on initial load — avoid speculative preconnects for origins only used after user interaction (those should remain `dns-prefetch`).

**Files affected:**
- `src/app/layout.jsx` — update `<head>` resource hints

**User impact:** Not visible. Reduces time-to-first-byte for GitHub contribution data fetch. No regression risk.

- **Per-project og:image:** Each project getting its own OG image for when someone shares a project modal URL. The site currently has no per-project routes, so this is N/A until project detail pages are added.
- **Sitemap.xml:** Not present. Adding it is a one-file change but is a standalone SEO task. Deferred to v0.3.0.
- **robots.txt improvements:** Current file (if any) is not audited. Deferred.
- **Google Search Console verification:** The layout has `verification: { google: 'your-google-site-verification-code' }` — a placeholder. Filling this in is an operational task, not a code task.
- **Animated GIF poster frames as static fallbacks for JavaScript-disabled browsers:** Edge case, deferred.
- **`@next/bundle-analyzer` integration:** Dependency is present in `package.json` but not configured. Useful for ongoing monitoring but not required for v0.2.0 fixes.
