# PRD: Portfolio v0.4.0 — Right Sidebar, Rich App Content, and Website Mode

**Status:** Ready
**Date:** 2026-04-09

---

## Version Goal

Elevate the desktop OS experience to PostHog-level polish by adding a persistent right sidebar with quick-launch links, upgrading three app components with real content (Mail with live contact API, Trash app, Gallery app), and introducing a Website Mode toggle that lets visitors escape the OS shell into a clean traditional portfolio layout.

---

## Background

v0.3.1 completed the visual identity: neutral dark theme, 11 color schemes, CSS variable system fully locked down. The shell is polished. What's missing is substance — the apps feel thin and the desktop feels empty. A visitor who doesn't know terminal commands has no clear path to the portfolio content.

This version addresses three gaps simultaneously:

1. **Discoverability** — a right sidebar (inspired by PostHog's) makes every section of the portfolio one click away without requiring any terminal knowledge.
2. **App depth** — Mail, Trash, and Gallery become real applications with meaningful content, not placeholders.
3. **Accessibility** — some visitors simply want a normal portfolio. Website Mode gives them one without a separate URL.

**What must not regress:**
- SEO pages (`/projects`, `/blog`, `/contact`) remain crawlable direct URLs
- All v0.3.1 theme system behavior — 11 themes, localStorage persistence, `--dt-*` variable architecture
- Window store: drag, resize, minimize, maximize, z-index stack, position persistence
- Boot overlay, terminal commands, context menu
- Contact form email delivery via nodemailer (used on the `/contact` SEO page)

## Features

### Feature 1: Right Sidebar

**Description:** A persistent vertical panel anchored to the right edge of the desktop canvas (not inside a window). Contains quick-launch links for the six primary navigation targets. Can be collapsed to a narrow icon rail and re-expanded. State persisted in Zustand + localStorage.

**Visual design:**
- Width: `200px` expanded, `40px` collapsed
- Position: `fixed right-0 top-0 bottom-48px` (top of viewport to above taskbar, leaving `48px` for taskbar)
- Background: `var(--dt-surface)` with `backdrop-filter: blur(16px)`
- Left border: `1px solid var(--dt-accent-border)`
- Transition: `width 200ms ease` on collapse/expand
- Z-index: `90` (above desktop icons at 10, below taskbar at 100, below context menu at 200, below windows at 20–89)

**Sidebar items:**

| Icon | Label | Action |
|------|-------|--------|
| 👤 | About Me | `openWindow('about')` |
| 📰 | Blog | `openWindow('logviewer')` |
| 📁 | Projects | `openWindow('filemanager')` |
| 📄 | Resume | `window.open('/resume.pdf', '_blank')` |
| ✉️ | Contact | `openWindow('mail')` |
| 🗑️ | Trash | `openWindow('trash')` |

Each item in expanded state: icon + label text, `48px` tall row. In collapsed state: icon only, centered. Hover state: `background: var(--dt-accent-border)`, `color: var(--dt-accent)`.

**Toggle button:** A small `◀` / `▶` chevron button at the top of the sidebar (below the top edge). Clicking it collapses/expands. In collapsed state, the chevron shows `▶`. `aria-label="Collapse sidebar"` / `"Expand sidebar"`.

**Desktop layout adjustment:** When the sidebar is expanded, the `DesktopIconGrid` area and window layer must not be obscured. The desktop padding is already `p-4` on the right — expand `paddingRight` to `216px` (200px sidebar + 16px gap) when sidebar is expanded. Windows opened while the sidebar is expanded should use a default position that avoids the sidebar — offset `defaultPos.x` to stay within the non-sidebar area.

**Zustand state** (new slice added to `useWindowStore` or a new `useSidebarStore`):
```js
sidebarOpen: true,            // default: open
toggleSidebar: () => void,
setSidebarOpen: (open) => void,
```
Persisted to `localStorage` under key `portfolio_sidebar_open`.

**Website Mode integration:** The sidebar is visible in OS mode only. In Website mode it is hidden entirely (see Feature 3).

### Feature 2a: Mail App — Live Contact Form

**Description:** The Mail app's Compose panel already has a working form UI (From, Subject, Message fields, Send button). Currently `handleSend` uses a fake `setTimeout` that always succeeds. This upgrade replaces the fake send with a real API call to a new `/api/contact` route that uses the existing `nodemailer` transport in `lib/email.js`.

**Current state:** `Mail.jsx` `handleSend` fires a `setTimeout(2000)` then shows success. No email is actually sent.

**What changes:**

1. New API route at `src/app/api/contact/route.js`:
   - Accepts `POST` with JSON body `{ name, email, subject, message }` (4 fields — matches both Mail.jsx and ContactInteractive.jsx)
   - Validates: `email` required + valid format, `message` required, `name` and `subject` optional
   - Calls `sendContactMessage({ name, email, subject, message })` — a new function added to `lib/email.js`
   - Returns `{ success: true }` on success, `{ error: "..." }` on failure
   - Rate limiting: none in v0.4.0 (deferred — this is a portfolio, not a production service)

2. New `sendContactMessage({ name, email, subject, message })` function in `lib/email.js`:
   - Uses the existing `transporter` (same SMTP config: `SMTP_SERVER`, `EMAIL`, `EMAIL_PASSWORD` env vars)
   - **Must check env vars inside the function body** before calling `transporter.sendMail()` — follow the same guard pattern as `sendVisitorAlert` (the transporter is created at module scope with potentially undefined values)
   - Sends to the owner address from `portfolio.json` `contact.email` (single source of truth — do NOT hardcode)
   - Subject: `Portfolio Contact: ${subject || '(no subject)'}` with `replyTo: email`
   - Body includes `name`, `email`, `subject`, `message`, and timestamp
   - Gracefully handles missing SMTP config: logs error, returns `{ success: false, error: 'Email not configured' }`

3. `Mail.jsx` `handleSend` updated:
   - Replace `setTimeout` mock with `fetch('/api/contact', { method: 'POST', body: JSON.stringify(form) })`
   - On network or API error: show error state inline (below the Send button): `"Failed to send. Try emailing directly: <email from portfolio.json>"` in `var(--dt-accent)` text — read email from `portfolio.json` contact data, do NOT hardcode
   - On success: existing success UI (checkmark, "Message sent!") remains the same
   - Send button shows `⏳ Sending...` with `disabled` during the fetch (already implemented)

**User-facing impact:** Visitors who compose a message in the Mail app now actually send an email to the portfolio owner. The `mailto:` fallback shown in the Inbox welcome email remains as backup.

**Files:**
- `src/app/api/contact/route.js` (new)
- `src/lib/email.js` (modified — add `sendContactMessage`)
- `src/components/apps/Mail.jsx` (modified — replace mock send with real fetch)

**Grid spec:**
- 3 columns, `gap: 12px`, `padding: 16px`
- Each card: thumbnail image (16:9 ratio, `object-fit: cover`), title below in `11px` monospace `var(--dt-text-muted)`
- Hover: thumbnail gets `border: 1px solid var(--dt-accent)` + `box-shadow: 0 0 12px var(--dt-accent-glow-soft)`
- Thumbnail images: Use existing project images from `public/images/optimized/` (the `portfolio.json` projects already have `image` fields pointing to these paths)

### Feature 3: Website Mode Toggle

**Description:** A toggle that switches between "OS Mode" (current full-screen desktop experience) and "Website Mode" (traditional, scroll-based portfolio layout). Website Mode renders a clean, standard portfolio page using existing data from `portfolio.json`. The toggle is accessible from three places: the taskbar system tray, the desktop context menu, and a new keyboard shortcut `Ctrl+Shift+W`.

**State management:**
```js
// New Zustand store: src/store/uiStore.js
websiteMode: false,
toggleWebsiteMode: () => void,
setWebsiteMode: (val: boolean) => void,
```
Persisted to `localStorage` under key `portfolio_website_mode`. Default: `false` (OS mode).

**Mode switching behavior:**
- Switching OS → Website: Desktop canvas fades out (`opacity: 0, transition: opacity 200ms`), Website layout fades in
- Switching Website → OS: Website layout fades out, Desktop canvas fades in
- The URL does not change (both modes are on `/`)
- The switch is instantaneous in terms of data — no page reload, no route change
- On initial load: if `localStorage.getItem('portfolio_website_mode') === 'true'`, render Website mode immediately (skip OS mode entirely, prevent boot overlay)

**Website Mode layout — sections in order:**
```
┌──────────────────────────────────────────────────────────────┐
│  [Nav bar: logo left, links right: Projects Blog Contact]    │
├──────────────────────────────────────────────────────────────┤
│  HERO                                                        │
│  Hi, I'm Manish Singh Parihar                                │
│  Full Stack & AI Engineer                                    │
│  [View Projects]  [Get in Touch]  [Switch to OS Mode →]      │
├──────────────────────────────────────────────────────────────┤
│  ABOUT                                                       │
│  Bio text from portfolio.json                                │
│  Skills grid (languages, frameworks, tools)                  │
├──────────────────────────────────────────────────────────────┤
│  PROJECTS                                                    │
│  Grid of project cards (reuse ProjectCard component)        │
├──────────────────────────────────────────────────────────────┤
│  BLOG                                                        │
│  Grid of blog post cards (reuse BlogGrid component)         │
├──────────────────────────────────────────────────────────────┤
│  CONTACT                                                     │
│  Simple form: Name, Email, Message — POST to /api/contact   │
├──────────────────────────────────────────────────────────────┤
│  [Switch back to OS Mode ↑]  link in footer                  │
└──────────────────────────────────────────────────────────────┘
```

**Styling for Website Mode:**
- Uses `next-themes` dark/light toggle (already present in the project) — the standard Tailwind dark theme, NOT the `--dt-*` variables
- Responsive and mobile-friendly (the OS mode is desktop-only, but Website mode must work on mobile)
- Reuse existing components where possible: `ProjectCard`, `BlogGrid`, `ContactInteractive`
- The Website mode layout is a single `WebsiteMode.jsx` component that conditionally replaces the `Desktop.jsx` render tree

**Toggle entry points:**

1. **Taskbar system tray** — Add a `[≡]` or `[⊞]` icon button to the right section of the taskbar (between ThemePicker and wifi indicator). Clicking it calls `toggleWebsiteMode()`. `title="Switch to Website Mode"`.

2. **Context menu** — Add a new menu item to `ContextMenu.jsx`:
   ```
   { label: 'Switch to Website Mode', action: () => toggleWebsiteMode(), icon: '🌐' }
   ```
   Place it above the last divider (before "View Source").

3. **Keyboard shortcut** — `Ctrl+Shift+W` anywhere on the desktop calls `toggleWebsiteMode()`. Add the listener in `Desktop.jsx` `useEffect`.

4. **Hero CTA in Website Mode** — "Switch to OS Mode →" button in the hero section (and footer link) calls `toggleWebsiteMode()`.

**Mobile behavior:**
- On viewport `< 768px`, the `Desktop.jsx` already renders `<MobileFallback />`. When `websiteMode === true`, skip `<MobileFallback />` entirely and render `<WebsiteMode />` instead. This gives mobile users the website layout automatically and makes it useful on phones.
- The `MobileFallback` component can be removed or repurposed: if `websiteMode === false`, show the current mobile fallback. If `websiteMode === true`, show `<WebsiteMode />` (which is already mobile-responsive).

**Boot overlay interaction:** The boot overlay only runs when `websiteMode === false`. The skip logic belongs in `DesktopPage.jsx` (the orchestration layer that owns boot state), NOT in `BootOverlay.jsx`. If `uiStore.websiteMode === true`, `DesktopPage.jsx` should skip rendering `<BootOverlay>` entirely and render `<WebsiteMode />` instead.

**Keyboard shortcut note:** The `Ctrl+Shift+W` listener should check `e.target.tagName` — do NOT toggle when the user is typing in an input/textarea (e.g., Mail compose, contact form).
