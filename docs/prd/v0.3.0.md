# PRD: Portfolio v0.3.0 ‚Äî Desktop OS Simulator

**Status:** Ready
**Date:** 2026-04-05

---

## Version Goal

Transform the portfolio from a scroll-page with a terminal widget into an interactive dark Linux desktop OS simulator, where every section of the portfolio lives inside a draggable, resizable app window on a persistent desktop canvas.

---

## Background

v0.2.0 completed performance and SEO groundwork (GIF-to-video conversion, H1s, og:image, canonical URLs). The site now loads fast and is indexable. v0.3.0 is the navigation paradigm shift: instead of a nav bar that routes to separate pages, visitors land on a dark cyberpunk desktop and open "apps" ‚Äî draggable windows ‚Äî to explore the portfolio.

This is inspired by PostHog's OS-style marketing page (warm, playful, earthy) but executed as the visual opposite: dark cyberpunk, terminal green, hacker aesthetic ‚Äî closer to a Kali Linux desktop or a sci-fi workstation than a friendly SaaS landing page.

The existing content (projects, blog posts, contact form, GitHub contributions) does not change. Only the shell ‚Äî navigation, layout, and how content is accessed ‚Äî is being rebuilt.

**What must not regress:**
- SEO: `/projects`, `/blog`, `/contact` must remain crawlable direct URLs
- JSON-LD structured data (Person, Website, ProfilePage schemas)
- Contact form email delivery (nodemailer)
- GitHub contributions ISR data fetch
- Analytics (PostHog + Google Analytics)
- Canonical URLs added in v0.2.0
- All video/media replacements from v0.2.0

## Design Specification

### Aesthetic

The palette is dark cyberpunk terminal, **not** macOS light, **not** Windows dark. Reference points:

- **Background:** Near-black with a very faint green or blue-green tint ‚Äî something like `#080d0a` (charcoal with a touch of forest). Not the pure `#09090b` currently in the Tailwind dark theme.
- **Primary accent:** Terminal green `#22c55e` (already used as `green-500` in the existing codebase ‚Äî keep it).
- **Secondary accent:** Cyan `#00e5ff` (the existing `--terminal-cyan` CSS variable, currently at `hsl(187 100% 50%)` in dark mode).
- **Surface:** Windows use a dark surface `rgba(10, 18, 12, 0.92)` with a subtle backdrop blur ‚Äî frosted dark glass, not opaque black.
- **Borders:** `1px solid rgba(34, 197, 94, 0.2)` ‚Äî faint green glow, not solid.
- **Text:** `#e8f5e9` (near-white with a very faint green tint) for primary, `#6b7a6e` for muted.
- **Wallpaper:** Dark background with a subtle animated grid or dot pattern (the existing `bg-dot-white/[0.2]` Tailwind class is already close ‚Äî keep it but slow the parallax scroll to near-zero since there is no scroll on the desktop).
- **Glow effects:** Green box-shadow glows on focused/active windows: `0 0 0 1px rgba(34, 197, 94, 0.4), 0 8px 32px rgba(34, 197, 94, 0.08)`.

### Window Chrome

macOS-style traffic light buttons (left side of titlebar, in order: red, yellow, green circles). These are the close / minimize / maximize controls. The existing codebase has no window chrome ‚Äî this is entirely new.

- **Close button:** `#ff5f57` (macOS red)
- **Minimize button:** `#febc2e` (macOS yellow)
- **Maximize/restore button:** `#28c840` (macOS green)
- **Titlebar text:** App name centered, monospace font, muted color
- **Titlebar height:** 36px, slightly frosted

### Fonts

No new fonts. The project already loads `IBM Plex Mono` and `Fira Code` ‚Äî both are perfect for the terminal aesthetic. Headlines inside windows use `IBM Plex Mono` at higher weights. The wallpaper layer uses `Fira Code` for any decorative text elements (e.g., fake system stats in a corner widget).

### 2. Window Manager

The core of the OS experience. Manages all open windows: their state, stacking order, drag/resize, minimize/maximize, and close.

**Window state per window:**

```
{
  id: string,
  appId: AppId,
  title: string,
  isOpen: boolean,
  isMinimized: boolean,
  isMaximized: boolean,
  position: { x: number, y: number },
  size: { width: number, height: number },
  zIndex: number,
  initialPath?: string   // for URL-driven auto-open
}
```

**Window manager operations:**
- `openWindow(appId, opts?)` ‚Äî open or focus-if-already-open
- `closeWindow(id)` ‚Äî remove from open windows list
- `minimizeWindow(id)` ‚Äî hide window, show as minimized in taskbar
- `restoreWindow(id)` ‚Äî un-minimize, bring to front
- `maximizeWindow(id)` ‚Äî expand to fill viewport minus taskbar height
- `focusWindow(id)` ‚Äî bring to front (increment global z-index counter, assign to this window)
- `setPosition(id, pos)` ‚Äî update position (called during drag)
- `setSize(id, size)` ‚Äî update size (called during resize)

**State management:** Zustand store. `zustand` is not currently in `package.json` ‚Äî it must be added (`bun add zustand`). This is the only new dependency required for the core window manager.

**Z-index strategy:** A global counter starts at 20. Each time a window is focused, the counter increments and the window receives the new value. No window ever needs a z-index above 200 (reserved for context menu) unless a very large number of windows are opened ‚Äî not a realistic concern for a portfolio.

**Drag and resize:** Two options evaluated:

- **Option A ‚Äî `react-rnd`:** A mature, well-tested library that handles both drag and resize in one component. `bun add react-rnd`. Simple to integrate ‚Äî wrap each window's content in `<Rnd>`. Handles edge snapping, bounds constraining to viewport. Recommended for v0.3.0.
- **Option B ‚Äî custom implementation:** Use pointer events + CSS transforms directly. More control, zero dependency, but significantly more code and edge cases (touch, resize handles, bounds).

**Decision:** Use `react-rnd` (Option A) for v0.3.0. This avoids implementing drag+resize from scratch, which has many edge cases. The library is 23KB gzipped and is client-side only.

**Bounds:** Windows must be constrained so their titlebar is always reachable (cannot drag a window's titlebar above `y=0` or below the taskbar).

**Window defaults (starting position and size per app):**

| App | Default position | Default size |
|-----|-----------------|--------------|
| Terminal | x: 80, y: 60 | 700 √ó 480 |
| File Manager | x: 120, y: 80 | 860 √ó 580 |
| Log Viewer | x: 160, y: 100 | 800 √ó 520 |
| Mail | x: 200, y: 80 | 760 √ó 560 |
| About | x: 240, y: 100 | 680 √ó 500 |
| Browser | x: 100, y: 60 | 900 √ó 600 |

**Files:**
- `src/store/windowStore.js` (new) ‚Äî Zustand store for all window state
- `src/components/desktop/WindowManager.jsx` (new) ‚Äî renders all open windows, provides context
- `src/components/desktop/Window.jsx` (new) ‚Äî individual window wrapper: titlebar with traffic lights, drag+resize via react-rnd, content slot, focus-on-click behavior
- `src/components/desktop/WindowTitlebar.jsx` (new) ‚Äî macOS-style titlebar with traffic light buttons and app title

### 3. Taskbar / Dock

Fixed at the bottom of the viewport. Height: 48px.

**Left section:** "Start" area ‚Äî shows a `>_` terminal logo / system name ("manish@portfolio:~$" styled text). Clicking it does nothing functional in v0.3.0 (future: could open a command palette).

**Center section:** App icons for all running/open windows. Each icon in the taskbar represents one open or minimized window. Clicking a taskbar icon:
- If window is minimized ‚Üí restore and focus it
- If window is open and focused ‚Üí minimize it
- If window is open and not focused ‚Üí focus it

Active/focused window gets a small green underline dot below its taskbar icon. Minimized windows get a muted/dimmed appearance.

**Right section:** System tray area ‚Äî clock (live, updates every second), a fake "wifi" icon, a fake "battery" icon (always at 87%, as an easter egg). These are decorative.

**Styling:** Taskbar background is `rgba(8, 14, 10, 0.85)` with `backdrop-filter: blur(12px)` and a `border-top: 1px solid rgba(34, 197, 94, 0.15)`. Slightly frosted, separated from the desktop by a faint green line.

**Files:**
- `src/components/desktop/Taskbar.jsx` (new) ‚Äî full taskbar component
- `src/components/desktop/TaskbarIcon.jsx` (new) ‚Äî individual app button in taskbar

### 4. Boot Sequence (First Visit)

On the very first visit (no `localStorage` key `portfolio_booted`), show a fullscreen boot sequence before the desktop appears. On subsequent visits, skip directly to the desktop.

**Boot sequence flow:**

1. Black screen, cursor blinks
2. BIOS-style text appears line by line (rapid typewriter):
   ```
   PORTFOLIO OS v0.3.0
   Copyright (C) 2026 Manish Singh Parihar
   
   Detecting hardware...............OK
   Loading kernel modules...........OK
   Mounting filesystems.............OK
   Starting network services........OK
   Loading portfolio data...........OK
   
   Welcome to manish@portfolio
   Type 'help' for available commands
   ```
3. Screen fades to the desktop
4. `localStorage.setItem('portfolio_booted', '1')` is set after fade completes

**Reuse opportunity:** The existing `BootSequence` component in `TerminalContext.jsx` shows a simplified boot inside the terminal. The new full-screen boot is a separate component that wraps the entire page. The existing `BootSequence` should be kept for use inside the Terminal app window.

**GSAP integration:** The existing `ClientWrapper.jsx` uses GSAP for entrance animations. The desktop boot transition should also use GSAP: the boot overlay fades out with `gsap.to(bootOverlay, { opacity: 0, duration: 0.8, onComplete: () => setBooted(true) })`. After `setBooted(true)`, the desktop icons stagger-animate in.

**Files:**
- `src/components/desktop/BootOverlay.jsx` (new) ‚Äî full-screen boot sequence component
- `src/store/bootStore.js` (new) or merged into `windowStore.js` ‚Äî tracks boot state

The routes `/projects`, `/blog`, `/contact` must remain functional as direct URLs for crawlers and for visitors who land via a shared link. These pages do NOT use the desktop OS ‚Äî they use the existing page layout (the terminal nav header, existing content components).

- `src/app/blog/page.jsx`

This is not a replacement for the canonical SEO routes ‚Äî it is an additional entry point for desktop-aware sharing.

### 8. Mobile Fallback

The OS desktop experience is fundamentally designed for screens ‚â•1024px wide. On smaller screens, show a simplified experience.

**Strategy:** CSS media query + conditional render in `Desktop.jsx`.

**Mobile experience (< 768px):** Do not render the desktop at all. Instead, render a simplified single-column layout showing:
1. A terminal-styled header (name, title, links)
2. The existing `Terminal.jsx` component (the terminal still works on mobile)
3. A grid of "app launcher" buttons (tappable icons with labels) that link to the SEO routes: `/projects`, `/blog`, `/contact`
4. GitHub contributions section

This is essentially the current v0.2.0 homepage, repurposed as the mobile fallback. The existing components (`Terminal`, `FeaturedProjects`, `GithubContributions`) are reused.

**Tablet (768px ‚Äì 1023px):** Show the desktop but with simplified window behavior ‚Äî windows open maximized by default, no drag/resize (too cramped). A single focused window at a time. The taskbar remains.

**Implementation:** The `Desktop.jsx` component uses `useMediaQuery` or a CSS approach to switch between:
- `>= 1024px`: Full OS desktop
- `768px ‚Äì 1023px`: Tablet mode (maximized windows only)
- `< 768px`: Mobile fallback (no desktop)

### 9. Window Position Persistence

On desktop (‚â•1024px), remember window positions and sizes using `localStorage`. When the user closes and reopens their browser, windows that were open reopen in their last known position.

**Scope:** Only persist position and size, not open/closed state. Windows do not auto-reopen on page load ‚Äî the user must open them again via icons or terminal commands. But when they do open, they start at their last saved position.

**Implementation:** Zustand's `persist` middleware with `localStorage` storage. Only serialize `position` and `size` per `appId` (not the full window object).

**Key:** `portfolio_window_positions`

---

### 10. Accessibility

Windows are interactive regions that must be keyboard-navigable.

- Each `Window` component renders as `role="dialog"` with `aria-label={title}`.
- When a window opens, focus moves to it (specifically to the first focusable element inside, or the close button if no focusable content).
- `Escape` key closes the focused window.
- `Alt+Tab` (or `Ctrl+Tab`) cycles focus between open windows ‚Äî this is a nice-to-have for v0.3.0; mark as stretch goal if time runs short.
- The taskbar buttons have `aria-label` and `aria-pressed` (for minimized/active state).
- Desktop icons have `role="button"` and respond to `Enter` and `Space` for keyboard activation.
- Context menu: `role="menu"` with `role="menuitem"` children. Arrow keys navigate items, `Escape` closes.
- The boot sequence has `aria-live="polite"` so screen readers announce the boot lines.

## Key Decisions Summary

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Drag+resize library | `react-rnd` | Mature, handles both drag and resize, well-tested, ~23KB |
| State management | Zustand | Already in CLAUDE.md preferred stack; not yet in package.json ‚Äî must add |
| Window chrome style | macOS traffic lights | User explicitly specified this preference |
| Terminal commands navigation | Open windows instead of routing | Keeps users in the OS experience instead of breaking to a page |
| SEO routes | Keep existing pages unchanged | `/projects`, `/blog`, `/contact` must remain crawlable |
| Mobile | Fallback to v0.2.0-style layout | Desktop OS on mobile is impractical |
| First-visit detection | `localStorage` key | Simple, no server state needed |
| App content loading | `dynamic()` (lazy) | Only Terminal is eager; all other apps load on first open |
| OG image | Keep static `og-image.png` from v0.2.0 | Dynamic generation deferred to a future version |
